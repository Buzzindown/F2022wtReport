<About-Relevant-Bits/>

<My-Job/>

  I was amazed by all of the tech that Shopify offers with its API and other developer tools. Alongside all of the neat Shopify stuff, I got to learn Koa which is just a lightweight version of Express. I was fortunate that everything was in ES6 TypeScript, since I had spent my last eight months working with React and JavaScript, all of the code was fairly easy to read. In particular I had never really worked with any backend code, so all of the server side tech was fascinating to me.

  The project I was assigned to was already partially developed but had been put aside for more than a year and was just now being revived in the hope that we could get it to the Shopify app store by the end of the semester. My biggest contribution to the project and the company by far was the development of the app’s analytics system. The app’s goal in short was to allow store owners to create customized, targeted material on their Shopify store based on visitor’s information and a set of “rules” that the store owner would create in the app. This is relatively useless if you have no way of determining the effectiveness of your customized marketing material and rules, so we needed a way to track and display this data in a meaningful way to the user. After the first month, and lots of discussion, the analytics dashboard and tracking became my project.

  My first challenge was trying to understand the code that was already in place, the code that was there seemed somewhat complete but it was still seriously broken and used some outdated tech. I took a few days to read through it and tried to understand the flow of the code that was already written. It gave me some ideas, but the main issue I arrived at was how this was seemingly going to become slower and slower as time went on and the database filled up. The flow at the time was essentially that we could track events in the user’s storefront, store them in the database and then query all of those analytics events for the store whenever the user wanted to view analytics. Since we were using DynamoDB, which utilizes NoSQL, there wasn’t any good way to structure and query a subset of this data. As time would go on, the query would get larger and slower.

  After much thought and lots of time spent at the drawing board with my senior developer, we came to a solution. In the frontend, we would allow users to query data for specific days, using a datepicker. This would limit the granularity of their analytics but it would allow us to create a smart querying system. For any set of dates that the user queried, we would check if the database already contained an entry related to those dates. The only way this could work is if we used very specific timestamps, in our case they would cover 24 hour blocks. Please see the diagram below as a visual representation of how these data structures would fit together.

  If we didn’t see one, then we would create an entry in the database which acts as a summary of all of the analytics events for that set of dates. So in the future, if the user queried this set of dates, they wouldn’t be querying the entire set of analytics events, they would just get back that small summary statistic almost immediately and the request would be much faster. This sped up queries that had already been requested at some point in the past and also limited cluttering in the database. For an idea of what this structure might look like please see the diagram below, which I had made up for our internal documentation.

  My next issue was trying to get analytics events to fire consistently in the frontend. Specifically the issue was that I couldn’t find a way to reliably fire http requests to the backend to store the analytics event data before the page unloaded. I basically needed a request that would fire off, and didn’t particularly care about getting a response back, since the page was unloading anyways. After asking around and reading lots, I was stumped, and a little frustrated but also very determined. Until one day I stumbled across the “sendBeacon” function, which is actually purpose built for sending analytics events before the page unloads. Now I was in business and the frontend and backend were coming together nicely. Prior to this term I hadn’t worked with any backend code, or really written anything end-to-end like this. I’m super thankful to have gotten to do this because it gave me such a good understanding and appreciation for all the moving parts that go into developing an application like this. We used Koa in the backend and a bunch of other new technologies that I had never seen before and I can’t put into words how much I have learned while working at Relevant Bits.

  My final challenge was to develop a full blown test suite. Which was also something I hadn’t faced before. The plan was to use Jest for unit tests, as well as frontend snapshot tests and then write an end-to-end test using Puppeteer with headless Chromium. I realized quickly that I wanted to interact with the database during testing rather than mocking everything for every test. Great, there’s an NPM package for that. I tried it, but it wasn’t exactly what we needed. I asked my peers if it would be reasonable to fork it and maintain our own version of the package, since most of what we needed was already there. After discussion, we concluded that it was an okay idea but we should try and find something else first. So, I ended up writing a script that would automate the creation and deletion of tables in our Dynalite (mock local database) instance. This allowed each test to have its own table during automated testing. This solution would scale with as many tests as we could possibly write. I was proud that I had done something like this because it was way out of my wheelhouse, it was scaleable, and it used technologies that I would’ve been anxious to use prior to this work term. From here, I finished out the analytics test suite, got it merged and continued to work on getting the app pulled together during the last three weeks of my work term.

<Reflections/>

  Now that you’ve read about my time at Relevant Bits, it’s time to do a little reflecting regarding my goals. As I had mentioned earlier these are just the broad goals that I had set at the start of the term and don’t fully represent the extent I went to planning and achieving other smaller goals over the course of the term.

  Regarding my first goal, which was about becoming better at talking about new ideas and solutions with my seniors. I think I achieved this goal. Initially I was quite nervous and shy despite everyone being more than friendly enough to make me feel comfortable. I think that once I became a little bit more established the pressure that I had been feeling before to perform was lifted, and I was less stressed while talking during meetings. This allowed me to have the confidence I needed to communicate my ideas fully. Another thing that I recognized was that I became more and more comfortable speaking while using very technical language to explain my ideas. Before this, I would often find myself using plain english to try and describe technical items because I would be in a bit of a panic and my brain couldn’t keep up. It felt awesome to convey my ideas in a way that other people could easily understand.

  In regard to my second goal which was about becoming more comfortable working with other people, particularly in a team setting. I also feel like I have achieved this goal. The social part of my goal was fairly easy to achieve, but that has little to do with me, and everything to do with my kind and caring peers. It was easy to get along with just about everyone and I didn’t feel nervous for very long at all. The part that was more difficult was the technical aspect, trying to read other people’s code and understand their ideas on the fly was challenging. It probably took me a good two to three weeks before I was able to easily read my peers’ code. Part of this is that everyone seems to have their own style, and it shows in the small areas of codes, like if they use ternary operators or not, or how they write their for loops. I think these things just take some time to learn and get comfortable with, but reading diverse styles of code has definitely improved my ability to quickly read and understand code in general.

   My final goal was in regards to continuing my use of creative thinking and problem solving. This goal turned out to be especially important this semester. I ended up working with a lot of new technologies this work term, and ended up revamping a lot of old code that was in the system. Because of these two factors, my options were sometimes fairly limited in how much I could change or write. Having said that, this actually made me become more creative as it put pressure on me to come up with clever solutions that worked within the constraints of the situation. Overall, I've definitely seen myself improve in this sense over the work term and would say that I have accomplished my goal.

  All in all, I’m quite pleased with what I’ve achieved this semester and hope to have equal or better success in achieving my goals during my following work terms.
